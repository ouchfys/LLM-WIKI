- 适用范围：当前 `FastAPI + Vue` 项目主线设计
- 关联文档：
  - `docs/design/DOCLING_UNIFIED_INGESTION_TASK.md`

## 2. 项目定位

### 2.1 产品名称

私有化贾维斯

### 2.2 一句话定义

一个面向求职、学习、论文阅读与知识沉淀场景的个人知识代理系统：它帮助用户采集外部内容、沉淀为可检索 Wiki，并基于用户画像从热门公开内容中做个性化筛选与推荐。

### 2.3 核心价值

系统的价值不在于“临时抓一篇文章并总结”，而在于把用户长期接触到的论文、面经、博客、项目、截图与笔记，持续沉淀为一个可对话、可检索、可演化的个人 Wiki。

## 3. 目标用户与使用场景

### 3.1 目标用户

- 求职中的算法、AI、基础模型、Agent 方向学生或工程师
- 需要持续阅读论文、GitHub 项目、技术博客的人
- 希望把零散信息沉淀成长期知识资产的人

### 3.2 核心场景

1. 用户在小红书、B 站、公众号、博客、GitHub、arXiv 上看到内容，想快速入库。
2. 用户读完一篇论文后，想生成结构化笔记并纳入自己的知识库。
3. 用户准备实习或面试时，希望直接和自己的知识库对话，而不是重新翻历史收藏。
4. 用户希望系统从“热门公开内容池”里做筛选，而不是被动等自己搜索。
5. 用户希望系统随着自己的使用过程逐步理解其方向、目标与当前关注点。

## 4. 产品原则

### 4.1 Markdown First

正文内容的最终沉淀形式是 Markdown，而不是数据库中的一段摘要字段。

原因：

- 人可读、可迁移、可导出
- 便于后续接入 Obsidian 风格展示或外部同步
- 便于 Agent 做增量编译、重写、链接与引用

### 4.2 Wiki First

用户感知到的是一个统一的个人 Wiki 产品，而不是多个互相割裂的子系统。

前台产品模型应统一为：

```text
内容采集 -> Wiki 沉淀 -> Wiki 对话 -> 个性化推荐 -> 精读升级
```

### 4.3 推荐优先做“过滤”，而不是做复杂预测

当前阶段推荐系统的目标不是工业级召回排序，而是从热门公开内容中，按用户目标和兴趣做有效过滤。

这比“凭空猜用户想看什么”更符合冷启动阶段，也更适合项目展示与面试讲述。

### 4.4 记忆服务于 Wiki 与推荐

记忆系统的目标不是做花哨的聊天人格，而是支撑两件事：

- 让 Wiki 对话具备连续性和上下文理解
- 让推荐系统能根据用户真实行为逐步变得更准

### 4.5 精读是可选能力，不是主入口

高成本的 PDF 深度解析、结构化问答、证据级追踪，应作为“精读模式”存在，而不是所有内容的默认处理路径。

## 5. 产品范围

### 5.1 当前主线范围

系统围绕五个核心能力展开：

1. 统一采集
2. Wiki 沉淀
3. Wiki Copilot 对话检索
4. 热门内容推荐与筛选
5. 可选精读

### 5.2 支持的主要内容类型

- 论文 PDF
- arXiv 论文链接与元数据
- GitHub 项目
- 技术博客/公众号文章
- 小红书链接与分享文本
- 用户上传图片
- 用户直接粘贴的文本笔记

### 5.3 当前不作为主目标的内容

- 重型企业知识图谱平台
- 大规模多租户 SaaS 权限体系
- 复杂协同编辑
- 通用网页搜索引擎替代品
- 全自动无人审核采集与批量灌库

## 7. 核心模块设计

### 7.1 内容采集层

#### 目标

把不同来源的内容统一转换成可落盘、可索引、可继续加工的 Markdown。

#### 设计要求

- 统一入口，不让用户理解底层解析差异
- 原始内容保留来源与证据
- 支持后续重新编译、重建索引、重新摘要

#### 输入来源

- URL
- PDF 文件
- 本地图片
- 文本粘贴
- 未来可扩展 DOCX / PPTX / HTML

#### 输出结果

- 原始 Markdown 文件
- 标准化元数据
- 对应 Wiki 卡片或候选卡片

### 7.2 Markdown Vault

#### 角色

Markdown Vault 是系统的正文事实层。

建议目录结构：

```text
data/
  wiki/
    raw_sources/
      paper_pdf/
      xiaohongshu_note/
      image_note/
      pasted_note/
      web_article/
    papers/
    interview/
    concepts/
    methods/
    notes/
    projects/
```

#### 设计约束

- 每个已入库对象至少有一个 Markdown 文件
- 原始源文件与结构化 Wiki 页面可分离，但必须可回溯
- SQLite 只存索引、状态和摘要，不作为正文唯一来源

### 7.3 Wiki 编译层

#### 目标

把原始内容转换成适合长期沉淀的 Wiki 页面。

#### 页面类型

- `PaperPage`
- `InterviewQA`
- `ConceptPage`
- `MethodPage`
- `ProjectPage`
- `SourceNote`

#### 主要职责

- 生成标题、摘要、标签
- 识别来源级别
- 结构化正文
- 建立页面间关联
- 处理去重与更新

#### 去重策略

优先按以下维度判重：

1. 规范化 URL
2. 来源平台 ID
3. PDF 文件哈希
4. 标题近似匹配

同一篇论文、同一条帖子、同一条分享文本不应被重复入库。

### 7.4 统一检索层

#### 目标

让 Wiki 对话检索的是全文，而不是只搜标题和摘要。

#### 当前主设计

- `wiki_pages`：卡片元数据
- `wiki_chunks`：全文分块索引
- SQLite FTS：当前主检索能力

#### 关键原则

- 检索围绕 Wiki 全文展开
- 论文正文不应成为孤岛表结构
- 所有可对话内容最终都应进入统一 chunk 索引

这意味着论文、面经、OCR 图片文本、本地笔记，都应收敛到统一检索面，而不是各自维护一套单独搜索逻辑。

### 7.5 Wiki Copilot

#### 目标

提供一个“面向个人知识库”的对话式入口，让用户通过自然语言直接检索、追问、对比和复盘自己的知识库。

#### 主要能力

- 基于全文 chunk 检索回答
- 引用命中的 Wiki 页面和片段
- 支持多轮追问
- 在合适场景下把对话结果写回 Wiki 或写入记忆层

#### 与通用聊天机器人的区别

它不是一个泛化聊天界面，而是用户个人知识库的操作面板与检索入口。

### 7.6 推荐系统

#### 产品目标

从公开热门内容中，筛出更适合当前用户的阅读候选。

#### 推荐池

- GitHub 热门项目
- arXiv 新论文 / 热门论文
- 技术媒体和公众号提到的热点论文
- 后续可加入高质量博客或技术社区来源

#### 推荐逻辑

第一阶段采用：

```text
热门内容池
  + 规则过滤
  + 用户画像打分
  + 多样性控制
  + 简单反馈回写
```

#### 冷启动策略

冷启动不依赖复杂画像建模，直接从公开热门内容出发：

- GitHub star、fork、趋势榜
- arXiv 最新/热门
- 技术媒体高频提及论文

随后再用用户目标、兴趣方向、近期行为做过滤与重排。

#### 推荐反馈

用户行为作为显式或隐式反馈写回：

- 保存到 Wiki
- 标记已读
- 发起精读
- 忽略
- 打开查看

### 7.7 用户画像与记忆系统

#### 目标

支撑推荐过滤、Wiki 检索上下文和逐步个性化，而不是做过细的聊天风格调优。

#### 分层设计

1. 短期记忆
   - 当前会话历史
   - 支持多轮追问与上下文改写

2. 情节记忆
   - 近期讨论过什么
   - 最近保存了什么
   - 最近在看什么
   - 最近忽略了什么

3. 稳定画像
   - 当前目标
   - 主要兴趣方向
   - 重点关注主题
   - 当前阶段性短板

4. 反馈日志
   - 推荐点击/忽略/保存
   - 精读启动
   - Wiki 入库行为

#### 画像字段建议

当前只保留与产品主目标强相关的字段：

- `goals`
- `interests`
- `focus_topics`
- `weak_points`
- `recent_topics`
- `saved_entities`
- `ignored_entities`

不建议当前阶段继续扩展大量弱相关偏好字段，例如回答文风、引用样式、输出长度等。

### 7.8 精读能力

#### 定位

精读能力是“升级路径”，不是系统主工作流。

#### 适用内容

- 用户明确要深入理解的论文
- 需要证据级回答的技术文档
- 面试或项目展示中需要重点讲解的代表性论文

#### 当前建议

- 保留精读入口
- 保留已入库论文问答能力
- 不再让旧 GraphRAG 体系主导整个系统设计

如果后续继续保留 `GraphRAG` 或类似重型方案，也应严格限定在少量高价值文档上。

## 8. 数据设计

### 8.1 文件系统

主要存储正文内容与原始沉淀文件。

```text
data/
  wiki/
    raw_sources/
    papers/
    interview/
    concepts/
    methods/
    notes/
    projects/
  raw_sources/
    xiaohongshu/
    images/
  papers/
```

### 8.2 SQLite

主要存储索引、元数据、状态与用户行为。

核心表建议包括：

- `wiki_pages`
- `wiki_chunks`
- `sessions`
- `messages`
- `user_profile`
- `user_profile_signals`
- `recommendation_feedback`
- `reading_items`
- `ingestion_tasks`

### 8.3 事实源优先级

统一约束如下：

1. Markdown 正文为主事实源
2. SQLite 为索引与状态层
3. 原始二进制文件为回溯证据

## 9. 关键流程

### 9.1 内容入库流程

```text
用户输入 URL / PDF / 图片 / 文本
  -> 下载或保存原始内容
  -> 解析 / OCR / 清洗
  -> 生成 raw markdown
  -> 编译成 Wiki 页面
  -> 写入 markdown vault
  -> 更新 wiki_pages
  -> 更新 wiki_chunks
```

### 9.2 Wiki 对话流程

```text
用户提问
  -> 读取当前 session 短期历史
  -> 必要时做问题改写
  -> 检索 wiki_chunks
  -> 生成答案与引用
  -> 写入 messages
  -> 更新情节记忆与画像信号
```

### 9.3 推荐流程

```text
抓取热门公开内容
  -> 形成候选池
  -> ProfileBuilder 产出画像快照
  -> 对候选做规则过滤和打分
  -> 生成推荐列表
  -> 用户点击/保存/忽略
  -> 行为反馈写回
```

### 9.4 精读流程

```text
用户选择某篇高价值论文
  -> 启动深度解析或精读索引
  -> 展示进度
  -> 完成后提供证据级问答
  -> 关键结论可回写 Wiki
```

## 10. 技术选型原则

### 10.1 前端

- 当前采用 Vue，继续作为主展示层
- 界面定位应更接近“个人知识工作台”，而不是玩具式 Demo

### 10.2 后端

- FastAPI 作为统一 API 层
- 各系统能力以模块方式组织，不把业务逻辑塞进路由文件

### 10.3 解析与 OCR

- Docling 可作为增强解析器
- 必须允许优雅降级
- 服务端安装重依赖可以接受，但不能让整个系统被单点依赖绑死

### 10.4 检索

- 当前优先 SQLite FTS + chunk 检索
- 如后续规模增长，再考虑向量召回补充，而不是先行过度设计

## 11. 阶段性实施策略

### 11.1 第一阶段：统一知识沉淀

目标：

- 所有主要内容源都能沉淀为 Markdown
- 去掉论文内容孤岛
- 让 Wiki Chat 真正搜全文

重点：

- 入库统一化
- Markdown 落盘
- `wiki_chunks` 稳定工作

### 11.2 第二阶段：推荐系统成型

目标：

- 建立热门内容池
- 做用户画像过滤
- 跑通冷启动与反馈闭环

重点：

- GitHub / arXiv / 技术媒体来源适配
- 阅读推荐页完善
- 推荐理由可解释

### 11.3 第三阶段：记忆与画像增强

目标：

- 提升系统连续使用体验
- 让推荐与检索更懂用户当前阶段

重点：

- 会话记忆稳定化
- 情节记忆清洗
- 画像信号去噪

### 11.4 第四阶段：精读升级

目标：

- 为代表性论文提供更强问答与证据引用能力

重点：

- 精读任务管理
- 进度可视化
- 精读结果回写 Wiki

## 12. 非目标与边界控制

以下内容不应在当前阶段继续扩大投入：

- 为所有内容默认做重型 RAG
- 过早引入复杂知识图谱展示
- 过度细分回答风格和偏好参数
- 为单用户场景做工业级推荐系统架构
- 为演示效果增加与主价值无关的复杂功能

## 13. 风险与对应策略

### 13.1 内容解析质量不稳定

策略：

- 原始 Markdown 永久保留
- 支持重新编译
- 解析失败允许回退

### 13.2 推荐冷启动效果一般

策略：

- 直接从热门池开始
- 先把过滤做好，而不是强行做个性化预测

### 13.3 用户画像噪声过高

策略：

- 只保留高价值画像字段
- 优先采用显式信号和高置信行为
- 降低一次性对话推断权重

### 13.4 精读链路成本高

策略：

- 只对少量高价值文档开放
- 做异步任务和进度展示
- 不把它放在默认路径上

## 14. 面试展示价值

本项目适合作为以下能力展示：

- 多源知识采集与统一沉淀
- Markdown-first 的个人知识系统设计
- 统一全文检索与对话式知识库
- 用户画像驱动的轻量推荐系统
- 记忆系统如何服务实际产品闭环
- 为什么从重型 GraphRAG 主线收敛到更适合个人场景的架构

## 15. 结论

当前项目最合适的主线不是“做一个巨大的论文 RAG 系统”，而是构建一个围绕个人知识沉淀与个性化筛选的知识代理。

因此，系统的统一方向应明确为：

```text
用统一采集把外部内容沉淀成 Markdown Wiki，
用对话检索让用户真正使用这些知识，
用画像与反馈从公开热门内容中做个性化过滤，
再把少量高价值论文升级到精读模式。
```

这条路线更符合真实用户场景，也更适合作为项目和面试展示主线。
