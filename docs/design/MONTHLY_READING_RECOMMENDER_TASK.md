# 月度推荐阅读方案

## 1. 文档定位

本文件定义推荐系统的第一阶段产品方案，重点是“从热门公开内容中做个性化过滤”，而不是构建复杂的工业级推荐平台。

它是主设计文档 [PERSONAL_RESEARCH_AGENT_PLAN.md](</e:/Agent-learn/agent项目/agent项目/docs/design/PERSONAL_RESEARCH_AGENT_PLAN.md>) 的专项执行说明。

## 2. 目标

为用户生成一个稳定、可操作、可反馈的“本月推荐阅读”列表。

这个列表的来源不是完全凭空生成，而是基于：

- 热门 GitHub 项目
- arXiv 最新 / 热门论文
- 技术媒体和公众号提到的热点论文或项目

再结合用户当前的目标、兴趣和近期行为做过滤与重排。

## 3. 为什么这样设计

### 3.1 冷启动简单

对于单用户知识代理，最难的不是个性化，而是冷启动。

如果没有足够历史行为，最可靠的做法就是：

- 先拿公开热门内容做候选池
- 再用用户画像做过滤

这比一开始就做复杂推荐模型更稳，也更容易讲清楚产品价值。

### 3.2 更符合真实使用场景

当前用户最真实的需求不是“学完所有 AI 知识”，而是：

- 找到当前阶段值得看的内容
- 筛掉和自己方向不相关的内容
- 把看过的内容沉淀进个人 Wiki

### 3.3 便于和 Wiki 闭环

推荐系统不是独立页面而已，它要和 Wiki 形成闭环：

```text
推荐 -> 阅读 -> 保存到 Wiki -> 对话检索 -> 反馈 -> 下一轮推荐
```

## 4. 产品定义

### 4.1 用户看到的不是“搜索结果”，而是“候选清单”

推荐页应该是一个稳定的阅读工作台，而不是临时搜索结果页。

每个推荐项至少包含：

- 标题
- 来源
- 类型
- 简介
- 推荐理由
- 当前状态
- 操作按钮

### 4.2 状态设计

建议状态：

- `candidate`
- `opened`
- `saved`
- `read`
- `deep_read`
- `ignored`

这些状态本身就是用户反馈数据。

### 4.3 推荐项操作

每个推荐项至少支持：

- 打开原始链接
- 保存到 Wiki
- 标记已读
- 发起精读
- 忽略

## 5. 候选池设计

### 5.1 第一阶段候选源

优先从这些公开来源构建候选池：

1. GitHub 热门项目
2. arXiv 最新 / 热门论文
3. 技术媒体提到的热门论文或项目

### 5.2 来源定位

#### GitHub

适合：

- Agent 框架
- RAG 工具链
- 开源模型工程
- 训练 / 部署 / 推理项目

#### arXiv

适合：

- 新论文
- 综述
- 与用户目标方向匹配的研究内容

#### 技术媒体

适合：

- 帮助用户快速筛热点
- 提供“论文为什么值得看”的二次语境

但媒体内容本身不应无条件高于一手论文或官方项目。

## 6. 排序策略

### 6.1 第一阶段排序公式

建议采用轻量可解释方案：

```text
总分 =
  热度基础分
+ 目标匹配分
+ 兴趣匹配分
+ 近期主题匹配分
+ 保存/阅读行为相关分
+ 新颖性分
+ 多样性调整
- 明确负反馈惩罚
```

### 6.2 不做的事情

当前不需要：

- 协同过滤
- 复杂深度学习排序模型
- 大规模召回框架
- 很多无意义的偏好参数

### 6.3 推荐理由可解释

推荐页必须能显示简洁理由，例如：

- 与你近期关注的 `RAG evaluation` 相关
- 属于你当前目标 `实习面试准备`
- 来自本周高热度 GitHub 项目
- 你最近保存过相近主题的论文

## 7. 用户画像如何参与推荐

### 7.1 使用的画像字段

推荐系统当前只消费和主需求强相关的字段：

- `goals`
- `interests`
- `focus_topics`
- `weak_points`
- `recent_topics`
- `saved_entities`
- `ignored_entities`

### 7.2 弱点字段的正确含义

这里的 `weak_points` 不表示“所有不会的 AI 知识”，而表示：

- 对当前目标有价值
- 用户多次涉及
- 但理解仍不稳定的主题

比如：

- 面试准备阶段反复问 `RAG evaluation`
- 做 Agent 项目时总在 `memory design` 上卡住

### 7.3 不建议过度接入回答风格

诸如 `answer_style`、`citation_style`、`length_preference` 之类的字段，不应成为当前推荐系统的主要输入。

它们对当前产品价值贡献太小。

## 8. 冷启动方案

### 8.1 默认路径

当用户没有足够历史行为时：

1. 从 GitHub 热门和 arXiv 热门构造候选池
2. 根据用户初始目标做粗过滤
3. 根据用户明确兴趣做第一次重排

### 8.2 可选引导

首次进入推荐页时，可让用户补充最少量信息：

- 当前目标
- 想重点关注的方向
- 当前阶段身份

只要这三项，足够支撑第一阶段过滤。

### 8.3 冷启动阶段的非目标

不要求第一天就“完全懂用户”。

第一阶段的目标只是：

- 推荐内容别太离谱
- 用户能很快开始保存和反馈

## 9. 与 Wiki 的闭环关系

### 9.1 保存到 Wiki 是推荐系统核心出口

推荐项被保存后，应触发：

- 原始内容落盘
- Wiki 卡片生成
- 全文索引更新

### 9.2 已读总结可以反向增强画像

用户读完后写一句总结，本质上是高价值显式反馈。

这类内容可用于：

- 更新兴趣方向
- 更新关注主题
- 更新“已覆盖内容”

### 9.3 精读是少量升级路径

推荐项中只有少量高价值论文需要进入精读模式。

推荐系统不应该默认把候选论文都送去重型处理。

## 10. 数据模型建议

### `reading_items`

建议保留：

- `id`
- `source_id`
- `title`
- `url`
- `summary`
- `source_type`
- `source_level`
- `recommend_month`
- `status`
- `score`
- `reasons_json`
- `metadata_json`
- `created_at`
- `updated_at`

### 行为反馈

推荐系统需要把以下行为持续写回：

- 打开
- 保存
- 已读
- 忽略
- 发起精读

## 11. API 与前端要求

### 后端

- 提供候选抓取接口
- 提供当月推荐列表接口
- 提供推荐状态变更接口
- 提供行为反馈写回接口

### 前端

推荐页至少应包含：

- 本月推荐列表
- 推荐理由
- 状态过滤
- 快速操作入口

不建议把推荐页做成“又一个搜索页”。

## 12. 验收标准

完成第一阶段后，至少满足：

1. 用户能看到本月推荐列表。
2. 列表来源不是纯手填，而是来自公开热门内容池。
3. 推荐项能显示简要推荐理由。
4. 用户操作会回写状态与反馈。
5. 保存到 Wiki 后能进入统一检索面。

## 13. 非目标

当前阶段不追求：

- 行业级推荐精度
- 很复杂的个性化特征工程
- 多用户协同过滤
- 自动生成长篇 newsletter

## 14. 结论

推荐系统当前最合理的形态不是“猜用户想看什么”，而是：

```text
从热门公开内容里，
按用户当前目标和方向做一层高质量过滤。
```

这条路线冷启动简单，解释性强，也最适合和个人 Wiki 形成产品闭环。
