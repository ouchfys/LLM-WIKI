# 记忆、画像与会话方案

## 1. 文档定位

本文件定义系统中的会话记忆、长期记忆、用户画像与行为反馈结构。

它的目标不是塑造聊天人格，而是为以下两条主链路服务：

1. Wiki Copilot 的连续对话与检索增强
2. 推荐系统的个性化过滤与反馈闭环

## 2. 设计目标

系统需要具备一种“越用越懂用户”的感觉，但这个感觉必须来自真实能力，而不是表面话术。

因此，记忆系统要解决的是：

- 多轮对话不断档
- 知道用户最近在关注什么
- 知道用户保存过什么、读过什么、忽略过什么
- 能把这些信息用于下一次检索和推荐

## 3. 核心原则

### 3.1 记忆服务产品，不服务噱头

当前阶段记忆系统的主要价值是：

- 更好地检索用户个人 Wiki
- 更好地筛选推荐内容

不应把大量精力投入到回答语气、引用格式、长度偏好这种边缘问题上。

### 3.2 分层存储，避免全塞一张表

必须区分：

- 短期会话记忆
- 情节记忆
- 稳定画像
- 行为反馈

这些数据生命周期和使用场景都不同，混在一起只会越来越脏。

### 3.3 只保留高价值画像字段

当前阶段应尽量克制，只保留和产品主目标强相关的画像字段。

### 3.4 显式信号优先于弱推断

优先级建议如下：

1. 用户明确填写或明确表达
2. 用户真实行为反馈
3. 多次重复出现的稳定模式
4. 单次对话中的弱推断

## 4. 记忆分层设计

## 4.1 短期会话记忆

### 目标

支持多轮对话里的上下文理解，例如：

- “继续刚才那个”
- “和上一篇论文比较一下”
- “你说的那个结论是在哪张卡片里”

### 存储

- `sessions`
- `messages`

### 使用规则

- 每轮都落库
- 提示词只注入最近少量历史
- 默认保留最近 `3-6` 轮用于回答

这层记忆只服务当前对话，不直接参与长期推荐打分。

## 4.2 情节记忆

### 目标

记录用户近期做过什么、聊过什么、在意什么。

典型例子：

- 最近连续问了 Agent memory
- 保存过一篇 RAG evaluation 论文
- 忽略过一组偏模型训练的论文
- 把某条面经加入了个人 Wiki

### 存储

当前代码中仍有 `preference_memory` 命名残留，但语义上它应收敛为 `episodic_memory`。

### 特征

- 有时间性
- 可以有 TTL
- 适合做最近主题召回

## 4.3 稳定画像

### 目标

存放相对稳定、跨会话复用的用户状态。

### 当前建议保留字段

- `goals`
- `interests`
- `focus_topics`
- `weak_points`
- `negative_interests`

其中：

- `goals`：当前阶段目标，如实习面试、论文阅读、项目搭建
- `interests`：长期兴趣方向，如 RAG、Agent、Infra
- `focus_topics`：近期重点关注话题
- `weak_points`：与当前目标强相关、但理解仍不稳定的主题
- `negative_interests`：用户明确不想看的方向

### 不建议当前保留为核心字段的内容

- `answer_style`
- `citation_preference`
- `length_preference`
- `tone_preference`

这些字段可以留接口，但不应该成为记忆系统重点。

## 4.4 行为反馈

### 目标

把用户的真实操作变成推荐和画像更新依据。

### 高价值行为

- 打开推荐项
- 保存到 Wiki
- 标记已读
- 发起精读
- 忽略
- 删除 Wiki 卡片

这些行为比单轮闲聊里推断出的“偏好”更可信。

## 5. 画像提取策略

### 5.1 第一优先级：显式输入

例如用户直接说：

- 我现在主要准备 Agent 方向面试
- 最近重点看 RAG 和 memory
- 我对训练基础设施不太感兴趣

这类信息可直接进画像。

### 5.2 第二优先级：结构化行为

例如：

- 连续保存多篇某主题论文
- 多次点击某类 GitHub 项目
- 持续忽略某类内容

这类比聊天语句更稳定。

### 5.3 第三优先级：对话启发式提取

从 Wiki 对话中抽取主题和关注点是合理的，但要降权。

原因：

- 很多问题只是临时好奇
- 不应一次提问就永久改写画像

## 6. 记忆如何服务 Wiki Copilot

### 6.1 对话连续性

通过短期会话记忆，让 Copilot 能理解上下文。

### 6.2 查询改写

当用户说“这个方法”和“上一篇”时，可结合近期对话把问题改写成可检索的独立查询。

### 6.3 历史主题补全

如果近期连续讨论某主题，可在检索前作为辅助上下文注入。

但要控制力度，避免旧主题污染新问题。

## 7. 记忆如何服务推荐系统

### 7.1 推荐使用哪些信号

推荐系统优先消费：

- 当前目标
- 长期兴趣
- 最近主题
- 保存历史
- 忽略历史
- 当前阶段性短板

### 7.2 阶段性短板如何使用

只有当某个主题满足以下条件时，才应被当作 `weak_point`：

1. 与当前目标强相关
2. 用户反复涉及
3. 理解或表达明显不稳定

否则不要强行给用户打上“薄弱点”标签。

### 7.3 不做“学完所有 AI 知识”的错觉

推荐系统应围绕用户目标做过滤，而不是默认用户要补完所有知识树。

## 8. Profile Builder 的职责

`ProfileBuilder` 应作为统一画像快照构建器，负责把分散数据源聚合为可消费快照。

### 输入来源

- `user_profile_signals`
- `user_profile`
- `recommendation_feedback`
- `sessions/messages`
- 情节记忆表

### 输出快照

建议输出：

```text
ProfileSnapshot
  - goals
  - interests
  - focus_topics
  - weak_points
  - negative_interests
  - recent_topics
  - recent_episodes
  - saved_entities
  - ignored_entities
```

### 约束

- 快照面向“检索和推荐消费”
- 不要扩张成一份大而全的用户人格档案

## 9. 数据清洗建议

当前系统已有一定记忆能力，但中短期需要做三类清洗：

### 9.1 命名清洗

- `preference_memory` 应逐步迁移为语义更准确的命名

### 9.2 信号去噪

- 清理旧数据库里历史遗留的脏画像信号
- 降低单次弱推断权重

### 9.3 行为优先

- 让推荐更多依赖真实行为，而不是只依赖启发式抽取

## 10. 前端呈现建议

用户不需要看到“很多很学术的画像字段”，但需要感知到系统在记住他。

可以考虑轻量展示：

- 当前目标
- 近期关注主题
- 最近保存内容
- 推荐理由来源

不要做成复杂的画像管理后台。

## 11. 验收标准

当前阶段的记忆系统至少应达到：

1. Wiki 对话支持会话连续性。
2. 推荐系统能消费统一画像快照。
3. 用户保存、忽略、已读等行为会进入反馈层。
4. 系统能根据近期主题做一定程度的推荐过滤。
5. 画像字段保持克制，没有大面积无用偏好堆积。

## 12. 非目标

当前阶段不追求：

- 复杂人格模拟
- 非常细致的回答文风控制
- 长链路自治 Agent 规划系统
- 向量数据库式大规模记忆中台

## 13. 结论

这个项目里的记忆系统，最合理的定义是：

```text
一个为 Wiki 检索和推荐闭环服务的轻量用户状态系统
```

只要它能让系统更懂用户最近在看什么、存了什么、要准备什么，它就已经具备真实产品价值。
